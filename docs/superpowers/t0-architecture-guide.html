T0 Architecture Guide

A self-contained tour of the CMS Tier-0 codebase. Read top to bottom, ~5 hours per chapter.

Chapter 1 · Foundations

Setup

~5h reading time · By the end you'll know what T0 is, where it sits in CMS, what's in the two source roots, and the shape of the processing chain.

The CMS detector at the LHC produces particle-collision data continuously while the accelerator is running. The online data-acquisition system (DAQ + High-Level Trigger) reduces raw collisions into streamer files — binary files of events grouped by trigger stream — and writes them, plus rich metadata, into the online Storage Manager database. Tier-0 (T0) is the first offline workload-management agent that reads from the Storage Manager database and drives every offline processing step required to turn streamer files into archive-ready RECO datasets and calibration conditions. T0 is also where the operational decisions about acquisition era, processing site, stream-by-stream quotas, and conditions upload all live.

Vocabulary you need before going further: a run is one continuous data-taking interval (often a few hours); a lumi-section ("lumi") is a fixed sub-interval of a run, ~23 seconds long, used as the unit for accounting and bookkeeping; a stream is the trigger-defined output channel from the HLT (e.g. Express, PhysicsCommissioning, ParkingDoubleMuonLowMass0); a primary dataset ("PD") is the physics-defined slice of events written into a stream; a streamer file is the on-disk file of a stream's events for one or more lumis. Every concept in this guide attaches to one of those.

Operationally, T0 is a deployment of the CMS WMAgent stack on a small set of vocms*.cern.ch hosts under the cmst0 user, with code, scripts, and configs all sourced from this repository at /data/tier0/. There is no monolithic "T0 server" — there are agents (one MainAgent and zero or more HelperAgents) polling Oracle databases on a schedule.

The codebase splits into two source roots. src/python/T0/ is the pure-library half: configuration DSL, run-configuration translation, job splitters, closeout APIs, condition-upload code, and the entire Oracle DAO tree. src/python/T0Component/ is the runtime half: WMAgent harness components (Tier0Feeder and the stub Tier0Auditor) that subclass WMCore.Agent.Harness and register polling worker threads. The split exists so the libraries can be unit-tested and reused without booting an agent.

The chain you'll spend the rest of the guide unpacking, in one sentence: Storage Manager → (Tier0Feeder discovers the run, configures it, ingests streamer files) → Repack → Express → PromptReco → AlcaHarvest → ConditionUpload → archive. Everything that follows is detail filling in arrows and boxes.

D1.2 — WMAgent harness anatomy. Every WMAgent component in the CMS workload-management ecosystem follows this two-layer pattern. The component wrapper (here Tier0Feeder) subclasses Harness and exists only to register a worker thread; the actual work lives in a BaseWorkerThread subclass (here Tier0FeederPoller) whose algorithm() method runs every pollInterval seconds. WorkerThreadManager is supplied by WMCore and you almost never touch it. This pattern is reused by every other WMAgent component (RucioInjector, JobAccountant, …) — once you've internalised it for Tier0Feeder, you've internalised the rest.

D1.3 — T0 database topology. Tier0Feeder talks to as many as six Oracle databases on every tick. Solid arrows are mandatory connections — if any of T0AST, HLTConfDB, or StorageManagerDB is missing or unreachable, the agent will not start. Dashed arrows are optional: each is gated by an if hasattr(config, "<Name>Database"): block in the constructor (see the code excerpt below). T0AST is the only database T0 writes to in any volume — it holds the run/stream/dataset configuration, all the streamer-to-job bookkeeping, and the closeout state. The others are read-mostly inputs (HLT menu, streamer arrival notifications, CMSSW release announcements) plus a single cross-agent publishing channel (T0DataSvc).

This is the entire component wrapper. Its job: register one worker thread (Tier0FeederPoller) with the manager, at pollInterval seconds. That is it. Every line of "what T0 actually does" is in Tier0FeederPoller. The two-layer split (compare D1.2) keeps the WMAgent harness contract minimal and the algorithmic code unit-testable. Note that self.pollTime = 1 is a vestigial attribute; the real polling cadence is config.Tier0Feeder.pollInterval (typically 30s).

The poller's constructor is where the database topology of D1.3 lands in code. The first three connections (T0AST via the implicit myThread.dbi, then HLTConfDB, then StorageManagerDB) are unconditional — if any one's connect URL is missing the agent crashes here. The last three (SMNotifyDB, PopConLogDB, T0DataSvcDB) are wrapped in if hasattr(config, "<Name>Database"): blocks, matching the dashed arrows in D1.3. Notice the consistent pattern: build a DBFactory, call .connect(), build a DAOFactory with package="T0.WMBS", store both on self. Every later self.daoFactory(classname="…") call resolves into the per-package class tree under src/python/T0/WMBS/Oracle/ — Chapter 3 picks this up.

Deeper dive

Why is Tier0Auditor a stub?

Open src/python/T0Component/Tier0Auditor/Tier0AuditorPoller.py and you'll find that algorithm() does essentially nothing:

def algorithm(self, parameters = None):
    """
    _algorithm_

    """
    logging.debug("Running Tier0Auditor algorithm...")
    myThread = threading.currentThread()

    return

This is intentional. Registering a no-op WMAgent component reserves the namespace and configuration block (config.Tier0Auditor.*) so future audit logic can be slotted in without changing the deployment shape. In practice, the real "auditing" T0 does — late-workflow detection, paused-job triage, backlog scanning — happens in standalone scripts under bin/ (cmst0_late_workflows, cmst0_paused_jobs, cmst0_backlog_wma), not inside this component. If you ever see "Tier0Auditor" in operator chatter, it's almost certainly the bin/ scripts being meant.

Key takeaways

  • T0 is a polling agent on top of WMCore's WMAgent framework, deployed on a small fixed set of vocms* hosts under cmst0.
  • The codebase splits into src/python/T0/ (pure libraries, unit-testable) and src/python/T0Component/ (WMAgent harness components: Tier0Feeder active, Tier0Auditor stub).
  • Every WMAgent component is a thin Harness subclass that registers a BaseWorkerThread subclass; T0 follows this pattern exactly.
  • Tier0Feeder connects to up to six Oracle databases per tick: T0AST, HLTConfDB, StorageManagerDB are mandatory; SMNotifyDB, PopConLogDB, T0DataSvcDB are optional (gated by hasattr(config, …)).
  • The publishable T0 version (src/python/T0/__init__.py) and the deployed T0 version (pinned in bin/00_pypi_deploy_*.sh) drift independently. Always check both.
