T0 Architecture Guide

A self-contained tour of the CMS Tier-0 codebase. Read top to bottom, ~5 hours per chapter.

Chapter 1 · Foundations

Setup

~5h reading time · By the end you'll know what T0 is, where it sits in CMS, what's in the two source roots, and the shape of the processing chain.

The CMS detector at the LHC produces particle-collision data continuously while the accelerator is running. The online data-acquisition system (DAQ + High-Level Trigger) reduces raw collisions into streamer files — binary files of events grouped by trigger stream — and writes them, plus rich metadata, into the online Storage Manager database. Tier-0 (T0) is the first offline workload-management agent that reads from the Storage Manager database and drives every offline processing step required to turn streamer files into archive-ready RECO datasets and calibration conditions. T0 is also where the operational decisions about acquisition era, processing site, stream-by-stream quotas, and conditions upload all live.

Vocabulary you need before going further: a run is one continuous data-taking interval (often a few hours); a lumi-section ("lumi") is a fixed sub-interval of a run, ~23 seconds long, used as the unit for accounting and bookkeeping; a stream is the trigger-defined output channel from the HLT (e.g. Express, PhysicsCommissioning, ParkingDoubleMuonLowMass0); a primary dataset ("PD") is the physics-defined slice of events written into a stream; a streamer file is the on-disk file of a stream's events for one or more lumis. Every concept in this guide attaches to one of those.

Operationally, T0 is a deployment of the CMS WMAgent stack on a small set of vocms*.cern.ch hosts under the cmst0 user, with code, scripts, and configs all sourced from this repository at /data/tier0/. There is no monolithic "T0 server" — there are agents (one MainAgent and zero or more HelperAgents) polling Oracle databases on a schedule.

The codebase splits into two source roots. src/python/T0/ is the pure-library half: configuration DSL, run-configuration translation, job splitters, closeout APIs, condition-upload code, and the entire Oracle DAO tree. src/python/T0Component/ is the runtime half: WMAgent harness components (Tier0Feeder and the stub Tier0Auditor) that subclass WMCore.Agent.Harness and register polling worker threads. The split exists so the libraries can be unit-tested and reused without booting an agent.

The chain you'll spend the rest of the guide unpacking, in one sentence: Storage Manager → (Tier0Feeder discovers the run, configures it, ingests streamer files) → Repack → Express → PromptReco → AlcaHarvest → ConditionUpload → archive. Everything that follows is detail filling in arrows and boxes.

D1.2 — WMAgent harness anatomy. Every WMAgent component in the CMS workload-management ecosystem follows this two-layer pattern. The component wrapper (here Tier0Feeder) subclasses Harness and exists only to register a worker thread; the actual work lives in a BaseWorkerThread subclass (here Tier0FeederPoller) whose algorithm() method runs every pollInterval seconds. WorkerThreadManager is supplied by WMCore and you almost never touch it. This pattern is reused by every other WMAgent component (RucioInjector, JobAccountant, …) — once you've internalised it for Tier0Feeder, you've internalised the rest.

D1.3 — T0 database topology. Tier0Feeder talks to as many as six Oracle databases on every tick. Solid arrows are mandatory connections — if any of T0AST, HLTConfDB, or StorageManagerDB is missing or unreachable, the agent will not start. Dashed arrows are optional: each is gated by an if hasattr(config, "<Name>Database"): block in the constructor (see the code excerpt below). T0AST is the only database T0 writes to in any volume — it holds the run/stream/dataset configuration, all the streamer-to-job bookkeeping, and the closeout state. The others are read-mostly inputs (HLT menu, streamer arrival notifications, CMSSW release announcements) plus a single cross-agent publishing channel (T0DataSvc).

The poller's constructor is where the database topology of D1.3 lands in code. The first three connections (T0AST via the implicit myThread.dbi, then HLTConfDB, then StorageManagerDB) are unconditional — if any one's connect URL is missing the agent crashes here. The last three (SMNotifyDB, PopConLogDB, T0DataSvcDB) are wrapped in if hasattr(config, "<Name>Database"): blocks, matching the dashed arrows in D1.3. Notice the consistent pattern: build a DBFactory, call .connect(), build a DAOFactory with package="T0.WMBS", store both on self. Every later self.daoFactory(classname="…") call resolves into the per-package class tree under src/python/T0/WMBS/Oracle/ — Chapter 3 picks this up.

Deeper dive

Why is Tier0Auditor a stub?

Open src/python/T0Component/Tier0Auditor/Tier0AuditorPoller.py and you'll find that algorithm() does essentially nothing:

def algorithm(self, parameters = None):
    """
    _algorithm_

    """
    logging.debug("Running Tier0Auditor algorithm...")
    myThread = threading.currentThread()

    return

This is intentional. Registering a no-op WMAgent component reserves the namespace and configuration block (config.Tier0Auditor.*) so future audit logic can be slotted in without changing the deployment shape. In practice, the real "auditing" T0 does — late-workflow detection, paused-job triage, backlog scanning — happens in standalone scripts under bin/ (cmst0_late_workflows, cmst0_paused_jobs, cmst0_backlog_wma), not inside this component. If you ever see "Tier0Auditor" in operator chatter, it's almost certainly the bin/ scripts being meant.

Key takeaways

  • T0 is a polling agent on top of WMCore's WMAgent framework, deployed on a small fixed set of vocms* hosts under cmst0.
  • The codebase splits into src/python/T0/ (pure libraries, unit-testable) and src/python/T0Component/ (WMAgent harness components: Tier0Feeder active, Tier0Auditor stub).
  • Every WMAgent component is a thin Harness subclass that registers a BaseWorkerThread subclass; T0 follows this pattern exactly.
  • Tier0Feeder connects to up to six Oracle databases per tick: T0AST, HLTConfDB, StorageManagerDB are mandatory; SMNotifyDB, PopConLogDB, T0DataSvcDB are optional (gated by hasattr(config, …)).
  • The publishable T0 version (src/python/T0/__init__.py) and the deployed T0 version (pinned in bin/00_pypi_deploy_*.sh) drift independently. Always check both.

Chapter 2 · The Tier0Config DSL

Setup

~5h reading time · By the end you'll know how operator-edited Python configs in etc/ become rows in T0AST, and which DSL setters to reach for when.

An operator running T0 never writes SQL. They write Python — specifically, a single file under etc/ (the canonical examples are etc/ReplayOfflineConfiguration.py and etc/ProdOfflineConfiguration.py) that imports a small set of setter functions from T0.RunConfig.Tier0Config and calls them in sequence. The result is an in-memory Tier0Configuration object — a deeply-nested WMCore.Configuration.Configuration tree — that captures every operational decision: which runs to inject, which acquisition era to stamp, what the per-stream Repack and Express thresholds should be, where conditions should be uploaded.

The Tier0Feeder loads this file at startup (config.Tier0Feeder.tier0ConfigFile points at it; the WMAgent config.py tweak scripts in bin/ set that path). Then, every time the feeder discovers a new run via FindNewRuns, it calls RunConfigAPI.configureRun(tier0Config, run, hltConfig) — and that is where the in-memory tree is translated into INSERTs against T0AST. The same applies stream-by-stream via configureRunStream. Once the inserts complete, the run is "configured" — bookkeeping rows exist for every stream, dataset, and trigger, and per-stream Repack/Express config is stamped on T0AST. From then on, every other component (job splitters, closeout APIs, conditions) reads only from T0AST; the in-memory tree is no longer consulted for that run.

The DSL is imperative, not declarative — calling addDataset mutates the tree by side effect, and order matters (e.g. setHelperAgentStreams must be called before any helper agent attempts to read). The full schema is documented as a giant ASCII tree at the very top of Tier0Config.py; that docstring is the single canonical reference. Reading it once is the highest-leverage 30 minutes you'll spend on T0 — every operational lever is named there.

Why it matters: when an incident comes in saying "stream X is using the wrong CMSSW version" or "PromptReco for run N never started" or "conditions aren't uploading", the answer is almost always either (a) something in the operator's etc/ Python is wrong, or (b) something in configureRun didn't translate it the way you'd expect. Knowing the DSL in detail is what makes those tickets tractable.

D2.3 — configureRun sequence. Triggered once per run, the first time FindNewRuns sees that run number. The function fetches the HLT mapping (which streams exist, which triggers go to which dataset), walks the in-memory Tier0Config, and emits the inserts in a fixed order: streams → datasets → triggers → mappings → per-stream configs → run-global update. Idempotency is enforced at the DAO level (every Insert* DAO uses MERGE or has unique constraints), so re-running configureRun on an already-configured run is safe.

Deeper dive

extraStreamDatasetMap — when does it apply?

The HLT menu defines which trigger paths fire which streams and which datasets receive which paths. Sometimes operators need to add a stream-to-dataset mapping that is not in the HLT menu — typically for parking PDs or special-purpose calibration streams that are routed differently downstream than the menu would imply. setExtraStreamDatasetMap(cfg, mapping) stuffs a dict into tier0Config.Global.extraStreamDatasetMap; configureRun reads it after walking the HLT menu and inserts additional STREAM_DATASET rows. The mapping has the shape { "StreamName": [{"Dataset": "DatasetName", "Path": "HLT_…"}] } — the optional Path is what makes this powerful: it can constrain event selection at PromptReco time. If the path is omitted, the entire stream's contents flow into the dataset.

Why is the DSL imperative?

Because WMCore.Configuration.Configuration is an attribute-mutable object, and the existing CMS WMAgent ecosystem builds all its configs imperatively (look at any WMAgent config.py tweak script in bin/). The advantage is that operators can use ordinary Python (loops, conditionals, comments) inside the configuration — see for example the for pd in PARKING_PDS: blocks in production configs. The disadvantage is no static validation: typos in attribute names just create new attributes that nobody reads. The validate-config.yaml CI workflow exists to catch the most common form of this (gaps in dataset family numbering).

Key takeaways

  • The Tier0Config object is built imperatively in Python by the operator's etc/*.py file. It is not a static config file format.
  • The docstring at the top of Tier0Config.py is the canonical schema reference. Re-read it whenever the DSL surprises you.
  • RunConfigAPI.configureRun is the bridge: it walks the in-memory tree once per run and emits all the SQL INSERTs against T0AST.
  • Streams.Default is a silent fallback for any stream not explicitly configured; deep-copy semantics, with the original Default never mutated by callers.
  • Once configureRun has finished, the in-memory Tier0Config is no longer consulted for that run — every other component reads from T0AST.

Chapter 3 · The Tier0Feeder heartbeat

Setup

~6h reading time · By the end you'll know exactly what one tick of Tier0FeederPoller.algorithm does, what each DAO is for, and how Main/Helper agents partition the load.

Everything in T0 happens because Tier0FeederPoller.algorithm() is invoked. The WMAgent harness's WorkerThreadManager calls it every config.Tier0Feeder.pollInterval seconds (typically 30) on the worker thread. There is no event-driven architecture, no message bus, no triggers; just a polling loop that asks "is there anything new since last time?" against several Oracle databases and acts on whatever it finds. If you want to predict T0's behavior, you read this method.

The four phases of one tick, paraphrasing the file's docstring: (1) discover new data from the Storage Manager and inject it into T0AST; (2) populate run-config rows for any new runs; (3) set up subscriptions for new run/streams; (4) feed streamer files into the configured subscriptions. A few additional steps run after that for monitoring, conditions release, and PromptReco release. These are sequential, not concurrent — each phase finishes before the next begins, and a try/except around the inner blocks ensures one failed phase doesn't stop the next tick from running on the same data.

Why the same component owns both configuration (calling configureRun*) and data flow (feeding streamers): keeping the two transactionally close means a row's existence in RUN_STREAM_FILESET_ASSOC is always preceded by the matching STREAM_DATASET_ASSOC rows in the same Oracle session. Splitting these into two pollers would have meant inventing a wait-for-config gate. The single-component design is simpler.

The DAO convention is uniform: every SQL touch is a class in src/python/T0/WMBS/Oracle/<area>/<Verb><Noun>.py, with a class body that holds a sql = """…""" string and an execute(...) method. There is no raw SQL anywhere in the poller — the poller only ever calls self.daoFactory(classname="…")(...).execute(...). This is the WMCore pattern; once you've read three DAOs, you've read them all. We sample a few below.

Finally, the Main vs Helper agent split: T0 in production runs as one MainAgent on one host and zero or more HelperAgents on other hosts. They share a Tier0Config (so they agree on which streams exist) but partition the workload by stream name: each helper agent processes only the streams listed for its name in tier0Config.Global.HelperAgentStreams[helperName]; the main agent processes everything not claimed by a helper. The mechanism is filter-based: same DAO/splitter code paths, just different filterStreamerFiles and filterHltConfigStreams implementations.

from T0Component.Tier0Feeder.MultipleAgents.BaseAgent import BaseAgent

class MainAgent(BaseAgent):
    """
    Main agent. The streams processed by this agent are implicitely defined in the tier0Config
    * This agent will process streams that are not defined in any of the other agents
    * tier0Config.Global.HelperAgentStreams = {'SecondAgent' : ["stream_1", "stream_2"],
                                                 'ThirdAgent' : ["stream_3", "stream_4"]}
    Note: This agent will ignore all streams in the helper agents, regardless of the helper agents status
    """
    def __init__(self, tier0Config):

        BaseAgent.__init__(self, tier0Config)
        self.name = "MainAgent"
        self.findUnwantedStreams()
        logging.info("This is a MainAgent. Ignoring streams {}".format(self.rejectStreams))

    def filterStreamerFiles (self, streamerFiles):
        """
        _filterStreamerFiles_

        When running multiple agents, filter out data from Storage Manager to avoid duplicates
        Returns the streamers the agent want to process
        """
        filteredStreamers = [streamer for streamer in streamerFiles if streamer['stream'] not in self.rejectStreams]

        return filteredStreamers

    def filterHltConfigStreams (self, hltStreamMapping = {}):
        """
        _filterHltConfigStreams_

        When running multiple agents, populate databases with the streams information relevant to the given agent
        """
        filteredHltStreamMapping = {stream: hltStreamMapping[stream] for stream in hltStreamMapping if stream not in self.rejectStreams}

        return filteredHltStreamMapping

    def findUnwantedStreams (self):
        """
        _findUnwantedStreams_

        Returns a list of streams that will not be processed by any other agent
        """
        self.rejectStreams = []
        for agent in self.helperAgentStreams.keys():
            self.rejectStreams += self.helperAgentStreams[agent]

MainAgent in full. The interesting line is findUnwantedStreams: it builds one flat list of every stream claimed by any helper, then both filter methods use a not in self.rejectStreams check. That avoids both agents racing on the same data when a helper restarts; the price is that streams claimed by a dead helper get stuck. (This is why the cmst0_late_workflows bin script exists.)

Deeper dive

What stops Main and Helper from racing on the same row?

Two layers. (1) Filter-based partitioning: as shown above, MainAgent and HelperAgent can never agree to process the same stream — Main's reject list is the union of all helpers' claim lists, and a helper's claim list is its only allowed input. (2) Row-level uniqueness in T0AST: every Insert* DAO either uses MERGE or has a unique constraint on the natural key (e.g. STREAM(stream_id, name)). If a race did somehow occur, the loser would hit a constraint violation rather than corrupt the row. (3) The optional SetDeploymentID DAO writes a per-deployment ID into a few rows so cross-deployment debugging can identify which agent wrote what.

Why is Tier0Config reloaded every tick instead of cached?

So that operators can edit etc/ReplayOfflineConfiguration.py live during a replay and see the change applied on the next tick (typically within 30 seconds). The cost is a small file read + Python parse per tick, which is negligible compared to the Oracle round trips. If the file is malformed the try/except around loadConfigurationFile falls back to "skip configuration but still process already-known runs" — meaning a bad config is non-destructive. (This is also why the validate-config.yaml CI workflow exists for production configs: production cannot self-heal in the same way without operator intervention.)

Key takeaways

  • Every tick: load Tier0Config → choose agent type → discover/inject → configure → feed → release → upload → mark monitoring. Sequential, idempotent, predictable.
  • All SQL lives in DAOs under src/python/T0/WMBS/Oracle/<area>/; the poller is pure Python orchestration. The classname-to-file mapping is one-to-one.
  • The Main/Helper split is configured in the Tier0Config (HelperAgentStreams) but selected at runtime by the WMAgent config's agentName. The two configurations must agree.
  • A misconfigured HelperAgent does nothing rather than ingesting random data — defensive emptiness is on purpose.
  • "New" is a property of T0AST state (e.g. RUN.acq_era IS NULL), not a property of the agent's memory. Restarting the agent re-discovers everything that's still in that state.
