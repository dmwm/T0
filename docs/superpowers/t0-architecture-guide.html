Chapter 1 · Foundations

Setup

~5h reading time · By the end you'll know what T0 is, where it sits in CMS, what's in the two source roots, and the shape of the processing chain.

The CMS detector at the LHC produces particle-collision data continuously while the accelerator is running. The online data-acquisition system (DAQ + High-Level Trigger) reduces raw collisions into streamer files — binary files of events grouped by trigger stream — and writes them, plus rich metadata, into the online Storage Manager database. Tier-0 (T0) is the first offline workload-management agent that reads from the Storage Manager database and drives every offline processing step required to turn streamer files into archive-ready RECO datasets and calibration conditions. T0 is also where the operational decisions about acquisition era, processing site, stream-by-stream quotas, and conditions upload all live.

Vocabulary you need before going further: a run is one continuous data-taking interval (often a few hours); a lumi-section ("lumi") is a fixed sub-interval of a run, ~23 seconds long, used as the unit for accounting and bookkeeping; a stream is the trigger-defined output channel from the HLT (e.g. Express, PhysicsCommissioning, ParkingDoubleMuonLowMass0); a primary dataset ("PD") is the physics-defined slice of events written into a stream; a streamer file is the on-disk file of a stream's events for one or more lumis. Every concept in this guide attaches to one of those.

Three more terms you'll see in every chapter and which deserve up-front definitions:

  • DAO (Data Access Object). A class in src/python/T0/WMBS/Oracle/<area>/<Verb><Noun>.py that holds one SQL statement and exposes an execute(...) method. T0 has no raw SQL anywhere outside DAO classes — every database touch goes through one. DAOs are loaded by classname ("Tier0Feeder.FindNewRuns") which resolves one-to-one to a file path. Chapter 3 covers the pattern in detail; for now, just know that "DAO" means "one SQL statement wrapped as a class".
  • WMBS (Workload Management Bookkeeping System). The Oracle schema WMCore uses to track all workload state — workflows, subscriptions, filesets, files, jobs. T0 extends WMBS with additional tables (run/stream/dataset config, conditions tracking) under the same Oracle instance. When you see package="T0.WMBS", that's the T0-specific DAO tree; package="WMCore.WMBS" is the shared WMCore one.
  • Replay vs production. Two distinct deployment modes. In production, T0 ingests real LHC collision data continuously as the StorageManager reports new streamer files. In a replay, T0 re-processes one or more historic runs identified by run number in setInjectRuns. Replays use etc/ReplayOfflineConfiguration.py + 00_pypi_deploy_replay.sh; production uses etc/ProdOfflineConfiguration.py + 00_pypi_deploy_prod.sh. Replays also use a different retry policy (PauseAlgo vs SquaredAlgo in production) and target different vocms hosts. The two modes share ~95% of the codebase — only the entry-point configs and a few tweak scripts differ.

Operationally, T0 is a deployment of the CMS WMAgent stack on a small set of vocms*.cern.ch hosts under the cmst0 user, with code, scripts, and configs all sourced from this repository at /data/tier0/. There is no monolithic "T0 server" — there are agents (one MainAgent and zero or more HelperAgents) polling Oracle databases on a schedule.

The codebase splits into two source roots. src/python/T0/ is the pure-library half: configuration DSL, run-configuration translation, job splitters, closeout APIs, condition-upload code, and the entire Oracle DAO tree. src/python/T0Component/ is the runtime half: WMAgent harness components (Tier0Feeder and the stub Tier0Auditor) that subclass WMCore.Agent.Harness and register polling worker threads. The split exists so the libraries can be unit-tested and reused without booting an agent.

The chain you'll spend the rest of the guide unpacking, in one sentence: Storage Manager → (Tier0Feeder discovers the run, configures it, ingests streamer files) → Repack → Express → PromptReco → AlcaHarvest → ConditionUpload → archive. Everything that follows is detail filling in arrows and boxes.

D1.2 — WMAgent harness anatomy. Every WMAgent component in the CMS workload-management ecosystem follows this two-layer pattern. D1.3 — T0 database topology. Tier0Feeder talks to as many as six Oracle databases on every tick. Solid arrows are mandatory connections — if any of T0AST, HLTConfDB, or StorageManagerDB is missing or unreachable, the agent will not start. Dashed arrows are optional: each is gated by an if hasattr(config, "<Name>Database"): block in the constructor (see the code excerpt below). T0AST is the only database T0 writes to in any volume — it holds the run/stream/dataset configuration, all the streamer-to-job bookkeeping, and the closeout state. The others are read-mostly inputs (HLT menu, streamer arrival notifications, CMSSW release announcements) plus a single cross-agent publishing channel (T0DataSvc).

D1.4 — T0 deployment topology. Where the code in this repo actually runs. The same code runs on all four vocms* hosts under the cmst0 Linux user, with the same /data/tier0/ directory layout. Differences between hosts are only the WMAgent config (which assigns each host's agentName as MainAgent, SecondAgent, etc.). Operators trigger replays via PR comment; the GitHub Action's self-hosted runner does the SSH-into-vocms work with a Kerberos keytab. Chapter 6 covers all this in detail.

The poller's constructor is where the database topology of D1.3 lands in code. The first three connections (T0AST via the implicit myThread.dbi, then HLTConfDB, then StorageManagerDB) are unconditional — if any one's connect URL is missing the agent crashes here. Every later self.daoFactory(classname="…") call resolves into the per-package class tree under src/python/T0/WMBS/Oracle/ — Chapter 3 picks this up.

Deeper dive

Why is Tier0Auditor a stub?

Open src/python/T0Component/Tier0Auditor/Tier0AuditorPoller.py and you'll find that algorithm() does essentially nothing:

def algorithm(self, parameters = None):
    """
    _algorithm_

    """
    logging.debug("Running Tier0Auditor algorithm...")
    myThread = threading.currentThread()

    return

This is intentional. Registering a no-op WMAgent component reserves the namespace and configuration block (config.Tier0Auditor.*) so future audit logic can be slotted in without changing the deployment shape. In practice, the real "auditing" T0 does — late-workflow detection, paused-job triage, backlog scanning — happens in standalone scripts under bin/ (cmst0_late_workflows, cmst0_paused_jobs, cmst0_backlog_wma), not inside this component. If you ever see "Tier0Auditor" in operator chatter, it's almost certainly the bin/ scripts being meant.

Key takeaways

  • T0 is a polling agent on top of WMCore's WMAgent framework, deployed on a small fixed set of vocms* hosts under cmst0.
  • The codebase splits into src/python/T0/ (pure libraries, unit-testable) and src/python/T0Component/ (WMAgent harness components: Tier0Feeder active, Tier0Auditor stub).
  • Every WMAgent component is a thin Harness subclass that registers a BaseWorkerThread subclass; T0 follows this pattern exactly.
  • Tier0Feeder connects to up to six Oracle databases per tick: T0AST, HLTConfDB, StorageManagerDB are mandatory; SMNotifyDB, PopConLogDB, T0DataSvcDB are optional (gated by hasattr(config, …)).
  • The publishable T0 version (src/python/T0/__init__.py) and the deployed T0 version (pinned in bin/00_pypi_deploy_*.sh) drift independently. Always check both.

Chapter 2 · The Tier0Config DSL

Setup

~5h reading time · By the end you'll know how operator-edited Python configs in etc/ become rows in T0AST, and which DSL setters to reach for when.

An operator running T0 never writes SQL. They write Python — specifically, a single file under etc/ (the canonical examples are etc/ReplayOfflineConfiguration.py and etc/ProdOfflineConfiguration.py) that imports a small set of setter functions from T0.RunConfig.Tier0Config and calls them in sequence. The result is an in-memory Tier0Configuration object — a deeply-nested WMCore.Configuration.Configuration tree — that captures every operational decision: which runs to inject, which acquisition era to stamp, what the per-stream Repack and Express thresholds should be, where conditions should be uploaded.

The Tier0Feeder loads this file at startup (config.Tier0Feeder.tier0ConfigFile points at it; the WMAgent config.py tweak scripts in bin/ set that path). Then, every time the feeder discovers a new run via FindNewRuns, it calls RunConfigAPI.configureRun(tier0Config, run, hltConfig) — and that is where the in-memory tree is translated into INSERTs against T0AST. The same applies stream-by-stream via configureRunStream. Once the inserts complete, the run is "configured" — bookkeeping rows exist for every stream, dataset, and trigger, and per-stream Repack/Express config is stamped on T0AST. From then on, every other component (job splitters, closeout APIs, conditions) reads only from T0AST; the in-memory tree is no longer consulted for that run.

The DSL is imperative, not declarative — calling addDataset mutates the tree by side effect, and order matters (e.g. setHelperAgentStreams must be called before any helper agent attempts to read). The full schema is documented as a giant ASCII tree at the very top of Tier0Config.py; that docstring is the single canonical reference. Reading it once is the highest-leverage 30 minutes you'll spend on T0 — every operational lever is named there.

Why it matters: when an incident comes in saying "stream X is using the wrong CMSSW version" or "PromptReco for run N never started" or "conditions aren't uploading", the answer is almost always either (a) something in the operator's etc/ Python is wrong, or (b) something in configureRun didn't translate it the way you'd expect. Knowing the DSL in detail is what makes those tickets tractable.

D2.2 — Setter → DAO → T0AST table mapping for the most-used DSL calls. Two important nuances: (a) the RUN row itself is created by StorageManagerAPI.injectNewData when a new run shows up in the StorageManager DB — not by any DSL call (setInjectRuns only puts a list of run numbers into the in-memory tree). D2.3 — configureRun sequence. Triggered once per run, the first time FindNewRuns sees that run number's row arrive (the row itself was inserted by StorageManagerAPI.injectNewData a few moments earlier). All six DAO executes happen inside one Oracle transaction; if any fails the whole thing rolls back. The order matters in code: UpdateRun runs first because the dataset/trigger inserts reference fields (era, backfill) that have to be in place. Per-stream configuration (InsertRepackConfig, InsertExpressConfig, InsertDatasetScenario) happens later in a different function — configureRunStream, called once per (run, stream) pair. Reco configuration (InsertRecoConfig) is later still, in releasePromptReco (Chapter 5). Idempotency is enforced at the DAO level, so re-running these is safe.

D2.4 — DSL setter timeline. The full operator-to-database lifecycle in one picture. Setters at the top run once, when the operator's Python file is loaded (which happens on every tick, but the resulting tree is identical until the operator edits the file). DAO inserts happen conditionally on what's new in T0AST — already-configured runs and run-streams are skipped, so a steady-state tick is mostly a no-op. The interesting consequence: editing etc/*.py mid-replay propagates to the next tick (~30 seconds), but only for runs/streams that haven't been configured yet. Deeper dive

extraStreamDatasetMap — when does it apply?

The HLT menu defines which trigger paths fire which streams and which datasets receive which paths. Sometimes operators need to add a stream-to-dataset mapping that is not in the HLT menu — typically for parking PDs or special-purpose calibration streams that are routed differently downstream than the menu would imply. setExtraStreamDatasetMap(cfg, mapping) stuffs a dict into tier0Config.Global.extraStreamDatasetMap; configureRun reads it after walking the HLT menu and inserts additional STREAM_DATASET rows. The mapping has the shape { "StreamName": [{"Dataset": "DatasetName", "Path": "HLT_…"}] } — the optional Path is what makes this powerful: it can constrain event selection at PromptReco time. If the path is omitted, the entire stream's contents flow into the dataset.

Why is the DSL imperative?

Because WMCore.Configuration.Configuration is an attribute-mutable object, and the existing CMS WMAgent ecosystem builds all its configs imperatively (look at any WMAgent config.py tweak script in bin/). The advantage is that operators can use ordinary Python (loops, conditionals, comments) inside the configuration — see for example the for pd in PARKING_PDS: blocks in production configs. The disadvantage is no static validation: typos in attribute names just create new attributes that nobody reads. The validate-config.yaml CI workflow exists to catch the most common form of this (gaps in dataset family numbering).

Key takeaways

  • The Tier0Config object is built imperatively in Python by the operator's etc/*.py file. It is not a static config file format.
  • The docstring at the top of Tier0Config.py is the canonical schema reference. Re-read it whenever the DSL surprises you.
  • RunConfigAPI.configureRun is the bridge: it walks the in-memory tree once per run and emits all the SQL INSERTs against T0AST.
  • Streams.Default is a silent fallback for any stream not explicitly configured; deep-copy semantics, with the original Default never mutated by callers.
  • Once configureRun has finished, the in-memory Tier0Config is no longer consulted for that run — every other component reads from T0AST.

Chapter 3 · The Tier0Feeder heartbeat

Setup

~6h reading time · By the end you'll know exactly what one tick of Tier0FeederPoller.algorithm does, what each DAO is for, and how Main/Helper agents partition the load.

Everything in T0 happens because Tier0FeederPoller.algorithm() is invoked. The WMAgent harness's WorkerThreadManager calls it every config.Tier0Feeder.pollInterval seconds (typically 30) on the worker thread. There is no event-driven architecture, no message bus, no triggers; just a polling loop that asks "is there anything new since last time?" against several Oracle databases and acts on whatever it finds. If you want to predict T0's behavior, you read this method.

The four phases of one tick, paraphrasing the file's docstring: (1) discover new data from the Storage Manager and inject it into T0AST; (2) populate run-config rows for any new runs; (3) set up subscriptions for new run/streams; (4) feed streamer files into the configured subscriptions. A few additional steps run after that for monitoring, conditions release, and PromptReco release. These are sequential, not concurrent — each phase finishes before the next begins, and a try/except around the inner blocks ensures one failed phase doesn't stop the next tick from running on the same data.

Why the same component owns both configuration (calling configureRun*) and data flow (feeding streamers): keeping the two transactionally close means a row's existence in RUN_STREAM_FILESET_ASSOC is always preceded by the matching STREAM_DATASET_ASSOC rows in the same Oracle session. Splitting these into two pollers would have meant inventing a wait-for-config gate. The single-component design is simpler.

The DAO convention is uniform: every SQL touch is a class in src/python/T0/WMBS/Oracle/<area>/<Verb><Noun>.py, with a class body that holds a sql = """…""" string and an execute(...) method. There is no raw SQL anywhere in the poller — the poller only ever calls self.daoFactory(classname="…")(...).execute(...). This is the WMCore pattern; once you've read three DAOs, you've read them all. We sample a few below.

Two WMCore concepts you'll see by name in this chapter and need to recognise: a fileset is a tracked collection of files in WMBS — one fileset corresponds to one logical input collection (e.g. "all streamer files for run N stream X"). A subscription is the pairing of a fileset with a workflow task (e.g. "the Repack task is subscribed to streamer fileset for run N stream X"). The splitters in Chapter 4 work over subscriptions: their job is to decide which files in the fileset to bundle into the next batch of jobs. FeedStreamers in this chapter is the DAO that adds new streamer rows to the relevant fileset.

Finally, the Main vs Helper agent split: T0 in production runs as one MainAgent on one host and zero or more HelperAgents on other hosts. They share a Tier0Config (so they agree on which streams exist) but partition the workload by stream name: each helper agent processes only the streams listed for its name in tier0Config.Global.HelperAgentStreams[helperName]; the main agent processes everything not claimed by a helper. The mechanism is filter-based: same DAO/splitter code paths, just different filterStreamerFiles and filterHltConfigStreams implementations.

Deeper dive

What stops Main and Helper from racing on the same row?

Two layers. (1) Filter-based partitioning: as shown above, MainAgent and HelperAgent can never agree to process the same stream — Main's reject list is the union of all helpers' claim lists, and a helper's claim list is its only allowed input. (2) Row-level uniqueness in T0AST: every Insert* DAO either uses MERGE or has a unique constraint on the natural key (e.g. STREAM(stream_id, name)). If a race did somehow occur, the loser would hit a constraint violation rather than corrupt the row. (3) The optional SetDeploymentID DAO writes a per-deployment ID into a few rows so cross-deployment debugging can identify which agent wrote what.

Why is Tier0Config reloaded every tick instead of cached?

So that operators can edit etc/ReplayOfflineConfiguration.py live during a replay and see the change applied on the next tick (typically within 30 seconds). The cost is a small file read + Python parse per tick, which is negligible compared to the Oracle round trips. If the file is malformed the try/except around loadConfigurationFile falls back to "skip configuration but still process already-known runs" — meaning a bad config is non-destructive. (This is also why the validate-config.yaml CI workflow exists for production configs: production cannot self-heal in the same way without operator intervention.)

Key takeaways

  • Every tick: load Tier0Config → choose agent type → discover/inject → configure → feed → release → upload → mark monitoring. Sequential, idempotent, predictable.
  • All SQL lives in DAOs under src/python/T0/WMBS/Oracle/<area>/; the poller is pure Python orchestration. The classname-to-file mapping is one-to-one.
  • The Main/Helper split is configured in the Tier0Config (HelperAgentStreams) but selected at runtime by the WMAgent config's agentName. The two configurations must agree.
  • A misconfigured HelperAgent does nothing rather than ingesting random data — defensive emptiness is on purpose.
  • "New" is a property of T0AST state (e.g. RUN.acq_era IS NULL), not a property of the agent's memory. Restarting the agent re-discovers everything that's still in that state.

Chapter 4 · JobSplitting

Setup

~5h reading time · By the end you'll know how streamer rows in T0AST become HTCondor jobs, and which DSL knobs drive each splitter's decisions.

A splitter is the function that takes a WMBS subscription (a fileset of input data attached to a workflow) and emits job groups — concrete units of work that JobSubmitter hands to HTCondor. Splitters do not run jobs; they decide which jobs to create, when, and with which inputs. Each phase of the T0 chain has its own splitter: Repack, RepackMerge, Express, ExpressMerge, Condition, AlcaHarvest. They all subclass WMCore.JobSplitting.JobFactory and implement a single method, algorithm(*args, **kwargs).

Discovery happens through WMCore's SplitterFactory plugin pattern: when JobCreator wants to make jobs for a subscription, it calls SplitterFactory(package="T0.JobSplitting") and the factory imports the module matching the subscription's splitAlgo attribute (e.g. "Repack" resolves to T0/JobSplitting/Repack.py). This is the same hook every WMAgent component uses — there's nothing T0-specific about the registration mechanism.

The four shapes you'll encounter:

  • Repack (Repack.py) — size-/lumi-/file-bounded. The biggest splitter, with the most knobs: maxSizeSingleLumi, maxSizeMultiLumi, maxInputEvents, maxInputFiles, maxLatency. Decides whether to split a single huge lumi into multiple jobs, accumulate small lumis into one job, or wait for more data.
  • RepackMerge (RepackMerge.py) — combines small Repack outputs into target-size files for storage. Same threshold style, different inputs (Repack outputs, not raw streamers).
  • Express (Express.py) — latency-bounded. Two thresholds (maxInputRate, maxInputEvents) plus an over-rate failure case. Designed to produce rapid feedback to detector experts; never waits if data is available.
  • ExpressMerge, Condition, AlcaHarvest — simpler shapes. ExpressMerge is usually a near-no-op; Condition produces calibration outputs; AlcaHarvest emits a single job per run/dataset. Skim them once, then return when needed.

The shared decision pattern across all of them: "wait for more, split now, or split-and-leave-some-behind". "Wait" is an active choice — if the splitter sees data but the thresholds aren't met, it returns no jobs; the next tick will see the same data plus whatever arrived since, and re-evaluate. There is no buffering inside the splitter; every decision is recomputed from current T0AST state.

Threshold values come from the Tier0Config DSL via configureRunStream, which reads tier0Config.Streams.<StreamName>.Repack.maxSizeSingleLumi (etc.) and writes them into per-stream rows in T0AST's config tables. By the time the splitter runs, those values arrive as kwargs on the algorithm call. So changing a threshold means: edit etc/*.py, wait for the next configureRun (next new run), and the new value applies.

Deeper dive

Why is RepackMerge a separate splitter instead of a flag on Repack?

Because in WMBS, a workflow is a directed graph of tasks, each of which runs over the outputs of its parent. Repack and RepackMerge are separate tasks with separate filesets and separate subscriptions. Combining them into one splitter would require a single subscription to ingest both streamers and Repack output files — incompatible with the subscription/fileset abstraction. By keeping them separate, each splitter has a single input shape, single threshold set, and produces a single output shape. The same applies to Express vs ExpressMerge: ExpressMerge consumes Express output, not streamers.

What does defineJobs in Express actually do with maxInputRate?

It iterates each lumi's streamers and computes lumiEventsTotal. If that total exceeds maxInputRate, the streamers for that lumi are passed to self.markFailed(lumiStreamerList) and the loop continues — no job is created for that lumi. Otherwise, streamers are packed into jobs respecting maxInputEvents: the first streamer always goes in; subsequent streamers are added only if the running event total stays under the cap. So a single huge streamer can still produce a job (it goes in alone), but a long pile of small streamers gets fan-out across multiple jobs. The implementation is in Express.py:70-140 if you want to read it.

Key takeaways

  • Splitters are pure decision functions: given current T0AST state and threshold values, decide which jobs to emit. No internal buffering.
  • Repack is size-driven with a latency override and a lumi-hole guard. Express is latency-driven with no waiting. The remaining splitters are simpler shapes.
  • Every threshold value originates in the operator's etc/*.py via the Tier0Config DSL; it reaches the splitter as kwargs on algorithm.
  • RepackMerge and ExpressMerge are separate splitters consuming the previous stage's output, not flags on Repack/Express.
  • Splitters produce jobs in WMBS state created; everything beyond is WMCore. The T0-specific retry policy (PauseAlgo on replays, SquaredAlgo on prod) decides what happens when a job fails repeatedly.

Chapter 5 · Closeout

Setup

~5h reading time · By the end you'll know exactly what has to happen for a run to go from "active" to "archived" — and where most "stuck run" tickets land.

"Closeout" is the convergence of four independent signals into the answer to one question: is run N completely done? Most operator triage tickets boil down to "this run won't close out — why?". The four signals come from four different subsystems, all driven by the same Tier0FeederPoller tick:

  1. PromptReco release (RunConfigAPI.releasePromptReco). PromptReco workflows for a run are deferred at run discovery — created in T0AST but not yet handed to JobCreator. They are released when (a) the relevant CMSSW release is announced (signal arrives via PopConLogDB) and (b) the configured RecoDelay for each dataset has elapsed since the run's stop time. Until both are true, no PromptReco jobs run for the run.
  2. AlcaHarvest (JobSplitting/AlcaHarvest.py). For each Express stream that produces calibration outputs, AlcaHarvest emits a single job that combines all the lumi-by-lumi alca outputs into one sqlite payload. Normally it waits for the input fileset to close (i.e. all Express jobs for the run are done). The global AlcaHarvestTimeout (set via setPromptCalibrationConfig on tier0Config.Global) is an escape hatch: if the run's reference time + timeout elapses, the splitter emits a partial-input job rather than waiting indefinitely. The reference time is the run's stop_time if it has been set; otherwise the splitter falls back to MAX(streamer.insert_time) for the run — the practical timing on replays where stop_time is 0 for much of the processing window.
  3. ConditionUpload (ConditionUpload/{ConditionUploadAPI.py,upload.py}). Once AlcaHarvest output (a sqlite + metadata pair) lands on EOS, the upload step copies it to the conditions dropbox over HTTPS. Two operational modes: validation mode writes to a side destination for human inspection; prompt mode writes to the live destination so PromptReco picks the conditions up immediately. The ValidationMode flag on the run controls which.
  4. RunLumi closeout (RunLumiCloseout/RunLumiCloseoutAPI.py). Marks lumis closed once their fileset has no more streamers, marks streams closed once all lumis are closed, and finally marks the run closed once every stream is closed and conditions are uploaded. Several "find active/stopped/closed" DAOs feed this — StopRuns sets stop_time, CloseRuns flips the run's close_time, etc.

The four signals are independent: each can complete in any order. The state machine doesn't advance until all have. So when an operator says "the run is stuck", the question is "stuck on which signal?" — and answering that question requires reading T0AST. Knowing this layer in detail is what makes such tickets tractable.

One more nuance: archival (the final step that frees up T0AST rows for cleanup) is owned by WMCore.WMComponent.TaskArchiver, not T0. T0's responsibility ends at closeout; archival happens N hours later (configurable via config.TaskArchiver.archiveDelayHours — 720 hours / 30 days in production).

This is also why agent restarts are non-destructive — the state lives in T0AST, not in the agent's memory.

Deeper dive

What does AlcaHarvestTimeout actually unblock?

Without a timeout, AlcaHarvest waits for the input fileset to close — which requires every Express job for the run to finish. If even one Express job is paused (e.g. on a transient error that exhausted retries), the fileset stays open and AlcaHarvest never runs. AlcaHarvestTimeout says: if a reference time + timeout is in the past, emit a partial-input AlcaHarvest job once (the previousAlcaHarvest guard prevents emitting more than one). The reference time comes from the GetRunStopTime DAO, which returns the run's stop_time if non-zero, else MAX(streamer.insert_time) for the run — important on replays, where stop_time is 0 for the bulk of processing. The cost of timing out: the calibration is computed from a possibly-incomplete set of Express outputs. The benefit: closeout proceeds. Operationally this is a global setting (not per-stream): setPromptCalibrationConfig(cfg, alcaHarvestTimeout=N, ...) writes tier0Config.Global.AlcaHarvestTimeout. Production typically uses a few hours.

ValidationMode vs prompt — what changes for the operator?

In validation mode, uploadToDropbox filters the destDB in the metadata file to point at the validation tag, not the production tag. The conditions still upload — they just land where physicists can inspect them before they're used by PromptReco. In prompt mode, the production tag is used, and PromptReco picks up the new conditions on its next workflow run. Replays default to validation; production defaults to prompt. The flag is set via setPromptCalibrationConfig(cfg, validationMode=True, ...) — the same setter that controls AlcaHarvestTimeout. There is no standalone setValidationMode function; both flags are tuned together because they're operationally coupled (the dropbox destination must match the prompt-calibration policy). The value is stamped on the run.valid_mode column at configureRun time.

Key takeaways

  • Closeout is the convergence of four independent signals: streams closed, AlcaHarvest done, conditions uploaded, CMSSW release announced. Missing any one stalls the run.
  • PromptReco release is signal-driven (waits for CMSSW announcement via PopConLogDB) and time-delayed (per-dataset RecoDelay after stop_time). Both must be true.
  • AlcaHarvest defaults to "wait for fileset close"; AlcaHarvestTimeout is the per-stream escape hatch when Express is stuck.
  • The T0 ↔ Conditions DB contract ends at the dropbox HTTPS POST. Anything past that is a different team's pipeline.
  • "Closed-out" and "Archived" are different states. Archival is owned by WMCore.WMComponent.TaskArchiver and happens hours-to-weeks later, not immediately.

Up to now we've treated T0 as a Python codebase. This chapter zooms out: T0 is also a small operating system. The deploy hosts (vocms047, vocms0500, vocms05011, vocms05012) all run under user cmst0, with everything orchestrated from /data/tier0/. So when an operator edits a deploy script on a vocms host without committing it back here, that change disappears the next time someone runs pypi_update.sh. The repo is the source of truth.

Each deploy script pins its own T0 and WMCore versions at the top. 00_pypi_deploy_replay.sh currently has WMAGENT_TAG=2.4.5 and TIER0_VERSION=3.5.5. 00_pypi_deploy_prod.sh may have different values. Drift between the two is normal: a new T0 version can be published days before a deploy script is updated to pull it in.

The single most operationally significant file in bin/ is 00_pypi_patches.sh. It is a plain shell script that curl | patchs WMCore PRs from contributor forks into the deployed virtualenv at deploy time. The patch URLs in that file are live operational state — they get rotated whenever a hotfix is needed, and the file as it sits in master is what the next deploy will apply. If you see surprising agent behaviour, read the patches file before reading the WMCore source on disk.

Operator UX for replays: the deployReplayPR.yaml GitHub workflow lets authorized operators trigger a replay deployment by commenting on a PR. The PR's job is to host the modified etc/*ReplayOfflineConfiguration.py file; the comment chooses the node, the T0/WMCore versions, the patches list, and the streams to run. The workflow runs on a self-hosted runner (runs-on: cmst0) with a Kerberos keytab — only authorized GitHub users (named in the workflow env.AUTHORIZED_USERS) can fire it; only whitelisted nodes (env.ALLOWED_NODES) can be targeted. This is why the replay flow can't be reproduced from a forked repo or external CI.

Finally, three small CI workflows backstop the codebase: validate-config.yaml gates PRs that touch any etc/*ProdOfflineConfiguration.py, running py_compile + a narrow flake8 rule subset + the dataset-family-gap detector at test/python/validate_config.py. create_tag_and_release.yaml tags+releases on changes to src/python/T0/__init__.py. pypi_build_publish_template.yaml publishes to PyPI on the same trigger.

Deeper dive

Why does the replay workflow run on a self-hosted runner (runs-on: cmst0)?

Because the workflow's deploy steps SSH into the target vocms host with Kerberos credentials. Those credentials live as a keytab on a CERN-side runner; pushing them into a public-cloud GitHub-hosted runner would expose them. So CMS provisions a small private runner (cmst0) inside the CERN network with the keytab on disk, and only that runner accepts the workflow. Side effect: a fork of this repo cannot reproduce the workflow — it has no runner labeled cmst0 available to it. Side effect 2: when the runner is offline, replays cannot deploy via the workflow (operators have to SSH manually).

What happens when __init__.py drifts from the deploy-script T0 pin?

Two independent things. (1) The published version (whatever's in __init__.py when CI fires) lands on PyPI as T0=={version}. (2) The deployed version (whatever's in 00_pypi_deploy_*.sh) is what the next deploy installs. If __init__.py says 3.5.6 but the deploy script still says TIER0_VERSION=3.5.5, then PyPI has both 3.5.5 and 3.5.6 but every fresh deploy still installs 3.5.5. To roll forward, you commit a PR bumping the deploy script's pin. To roll back, you do the same but to a lower version that's still on PyPI. Yanked versions (rare) require a different recovery path. This is the divergence that lets ops decouple "publishable" from "deployable" — useful when a version is on PyPI but operationally not yet ready.
